# CS 257 Compiler – Part 2: LLVM IR Builder

## Overview

This part implements the **LLVM IR code generation** phase of the compiler. It takes the AST produced by the Part 1 frontend and emits LLVM Intermediate Representation (IR), which can then be compiled to native machine code.

**Input:** A pointer to the root `astNode` of the AST (built by Part 1's parser)
**Output:** An `LLVMModuleRef` representing the full LLVM IR for the miniC program

The IR generator is split into two phases:

1. **Variable Renaming** – gives every variable a unique name to handle nested scopes and shadowing
2. **IR Generation** – traverses the AST and emits LLVM IR instructions into basic blocks

## Implementation

### Phase 1: Variable Renaming (`RenameCtx`)

Because miniC allows variable shadowing across nested scopes (e.g., two variables named `a` in different blocks), all variables must be renamed to unique names before LLVM allocas are generated.

A scope stack tracks all visible names at each nesting level:

```
scopes: [ function-scope | block-scope | nested-block-scope ]
```

Each scope is a map from the original name to its unique name. Unique names are generated by appending a counter suffix:

```
a  →  a$1   (first declaration)
a  →  a$2   (shadowing declaration in inner scope)
```

| Event                     | Action                                              |
|---------------------------|-----------------------------------------------------|
| Enter function            | Push new scope; rename parameter if present         |
| Enter nested block        | Push new scope                                      |
| Exit function / block     | Pop scope                                           |
| Declaration statement     | Generate unique name; insert into current scope     |
| Variable use (expression) | Look up original name in scope stack; rewrite node  |

After renaming, a set `allVarsUnique` contains every unique name that will need an alloca in the entry block. The parameter's unique name is saved separately as `paramUnique`.

### Phase 2: IR Generation (`IRGen`)

The IR generator uses the LLVM C API to construct an `LLVMModule`. It maintains a global `IRGen` struct holding:

| Field       | Type                                | Purpose                                            |
|-------------|-------------------------------------|----------------------------------------------------|
| `M`         | `LLVMModuleRef`                     | The LLVM module being built                        |
| `B`         | `LLVMBuilderRef`                    | The instruction builder (tracks insertion point)   |
| `curFn`     | `LLVMValueRef`                      | The function currently being generated             |
| `printFn`   | `LLVMValueRef`                      | Reference to the extern `print` declaration        |
| `readFn`    | `LLVMValueRef`                      | Reference to the extern `read` declaration         |
| `var_map`   | `map<string, LLVMValueRef>`         | Maps unique variable name → alloca instruction     |
| `ret_ref`   | `LLVMValueRef`                      | Alloca for the function's return value             |
| `retBB`     | `LLVMBasicBlockRef`                 | The shared return basic block                      |

#### Top-Level: `buildIR`

```
1. Create module "miniC", set target to "x86_64-pc-linux-gnu"
2. Add extern declarations for print and read
3. Rename all variables in the AST (Phase 1)
4. Add LLVM function with correct signature
5. Create entry basic block; position builder at end of entryBB
6. Emit alloca for every unique variable name → store in var_map
7. Emit alloca for return value slot → store in ret_ref
8. Emit store of parameter into its alloca (LLVMGetParam)
9. Create return basic block (retBB):
   - load from ret_ref
   - ret instruction
10. Generate IR for function body via genIRStmt(body, entryBB)
11. If exitBB has no terminator, emit unconditional branch to retBB
12. Prune unreachable basic blocks via BFS from entryBB
13. Dispose builder
```

---

### `genIRStmt` – Statement IR Generation

Takes a statement AST node and a `startBB` (the basic block to begin emitting into). Returns an `endBB` (the basic block that the next statement should continue from).

#### Assignment Statement
```
position builder at end of startBB
generate RHS value via genIRExpr
emit store(RHS, var_map[lhs_name])
return startBB
```

#### Call Statement (print)
```
position builder at end of startBB
generate argument value via genIRExpr
emit call to printFn with argument
return startBB
```

#### Return Statement
```
position builder at end of startBB
generate return expression via genIRExpr
emit store(expr, ret_ref)
emit unconditional branch to retBB
create and return a fresh endBB (continuation; will be pruned if unreachable)
```

#### Block Statement
```
set prevBB = startBB
for each statement S in the block's statement list:
    prevBB = genIRStmt(S, prevBB)
return prevBB
```
*(Declaration statements produce no IR; allocas were already emitted in entry.)*

#### If Statement
```
position builder at end of startBB
generate condition via genIRExpr  → i1 value
create trueBB and falseBB
emit conditional branch: cond ? trueBB : falseBB

if no else body:
    ifExitBB = genIRStmt(if_body, trueBB)
    if ifExitBB has no terminator: emit branch to falseBB
    return falseBB

if else body:
    ifExitBB   = genIRStmt(if_body,   trueBB)
    elseExitBB = genIRStmt(else_body, falseBB)
    create endBB
    if ifExitBB   has no terminator: emit branch to endBB
    if elseExitBB has no terminator: emit branch to endBB
    return endBB
```

#### While Statement
```
position builder at end of startBB
create condBB, bodyBB, exitBB
emit unconditional branch startBB → condBB

position builder at end of condBB
generate condition via genIRExpr  → i1 value
emit conditional branch: cond ? bodyBB : exitBB

bodyExitBB = genIRStmt(body, bodyBB)
if bodyExitBB has no terminator: emit unconditional branch bodyExitBB → condBB

return exitBB
```

---

### `genIRExpr` – Expression IR Generation

Takes an expression AST node and returns an `LLVMValueRef`.

| AST Node Type | Action                                                                                  |
|---------------|-----------------------------------------------------------------------------------------|
| `ast_cnst`    | `LLVMConstInt(i32, value)` → return constant                                           |
| `ast_var`     | `LLVMBuildLoad2` from `var_map[name]` → return loaded value                            |
| `ast_uexpr`   | Recurse on inner expr; `LLVMBuildSub(0, v)` → return negated value                     |
| `ast_bexpr`   | Recurse on lhs and rhs; emit `Add`/`Sub`/`Mul`/`SDiv` based on operator → return result |
| `ast_rexpr`   | Recurse on lhs and rhs; `LLVMBuildICmp` with matching `LLVMIntPredicate` → return `i1` |
| `ast_call`    | `read()` only — emit `LLVMBuildCall2` to `readFn` → return call result                 |

Relational operator mapping:

| miniC operator | LLVM Predicate  |
|----------------|-----------------|
| `<`            | `LLVMIntSLT`    |
| `>`            | `LLVMIntSGT`    |
| `<=`           | `LLVMIntSLE`    |
| `>=`           | `LLVMIntSGE`    |
| `==`           | `LLVMIntEQ`     |
| `!=`           | `LLVMIntNE`     |

---

### Dead Code Elimination

After generating IR for the function body, unreachable basic blocks are removed via **BFS** from the entry block:

```
visited = BFS reachability from entryBB
for each basic block BB in the function:
    if BB not in visited:
        LLVMDeleteBasicBlock(BB)
```

This cleans up the "after_return" continuation blocks created after `return` statements.


```bash
./ir_builder builder_tests/p1.c
```

Prints the generated LLVM IR to stdout.

```bash
make clean
```

## Example: Generated IR for `p1.c`

Input:
```c
extern void print(int);
extern int read();

int func(int p){
    int a;
    int b;
    a = 10;
    b = a + p;
    return b;
}
```

Output (`out/p1.ll`):
```llvm
declare void @print(i32)
declare i32 @read()

define i32 @func(i32 %0) {
entry:
  %"b$1" = alloca i32, align 4
  %"a$1" = alloca i32, align 4
  %"p$1" = alloca i32, align 4
  %ret    = alloca i32, align 4
  store i32 %0, ptr %"p$1", align 4
  store i32 10, ptr %"a$1", align 4
  %"a$1_ld" = load  i32, ptr %"a$1", align 4
  %"p$1_ld" = load  i32, ptr %"p$1", align 4
  %add      = add   i32 %"a$1_ld", %"p$1_ld"
  store i32 %add, ptr %"b$1", align 4
  %"b$1_ld" = load  i32, ptr %"b$1", align 4
  store i32 %"b$1_ld", ptr %ret, align 4
  br label %return

return:
  %rv = load i32, ptr %ret, align 4
  ret i32 %rv
}
```

---

## Implementation Notes

- All allocas are emitted together at the top of the `entry` block, regardless of where declarations appear in the source. This is standard practice for LLVM mem2reg optimization compatibility.
- The `return` basic block is always created upfront and shared by all return paths in the function, which simplifies control flow for functions with multiple exit points.
- Because `read()` can appear both as a statement (its return value is unused) and as an expression (its return value feeds an assignment), it is handled in both `genIRStmt` and `genIRExpr`.
